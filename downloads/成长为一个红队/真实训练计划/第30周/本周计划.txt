【本周知识清单】
- 模糊测试(Fuzzing)概念：
  - 定义：自动向程序输入大量随机/半随机数据，监控异常行为
  - 分类：Generation-based(基于生成) vs Mutation-based(基于变异)
  - 变异操作：位翻转、字节交换、算术运算、随机字节插入、字典替换
  - 覆盖引导(Coverage-Guided): 根据代码覆盖率反馈指导Fuzzing方向
- AFL (American Fuzzy Lop) 原理：
  - 编译时插桩：afl-gcc / afl-clang 在编译阶段插入覆盖测量代码
  - QEMU模式：无需源码，通过QEMU模拟执行进行Fuzzing
  - 基于突变+覆盖引导：变异输入，运行程序，记录新覆盖路径→保留
  - 种子(Seed): 初始输入(格式正确→更高效)
  - 崩溃去重和分类：AFL自动将崩溃分类为不同unique crash
- Fuzzing成果分析：
  - Triage崩溃：从AFL输出目录提取崩溃样本
  - 分析崩溃原因：GDB调试、ASAN(AddressSanitizer)报告
  - 判定可利用性：是否可控EIP/RIP？是否有保护机制？
  - Exploit开发：基于Fuzzing发现的崩溃开发完整利用链
- 综合考核要求：
  - 在VulnHub/HTB上选择一台未接触过的靶机
  - 完成全流程：侦察→扫描→漏洞发现→利用→提权→后渗透
  - 输出符合标准的渗透测试报告
  - 自我评估30周学习成果

📚 第1小时 | 理论学习 (09:00 - 10:00)
  学习内容：模糊测试原理与AFL基础
    - Fuzzing的基本概念：
      - 目标：在不了解程序内部实现的情况下，自动发现软件漏洞
      - 适用场景：处理复杂输入格式的程序(图片库、视频解码器、压缩库、网络协议)
      - 优势：自动化远超手动代码审计的效率
      - 局限：取决于Fuzzing质量和时间，无法保证所有漏洞都能被发现
    - AFL (American Fuzzy Lop) 架构：
      - 编译时插桩(Compilation-time Instrumentation):
        - afl-gcc/afl-clang替代标准编译器
        - 在汇编代码的基本块(Basic Block)入口插入覆盖测量代码
        - 通过共享内存位图(64KB bitmap)记录覆盖信息
      - 主循环(Fuzzing Loop):
        - 从队列取一个输入(种子)
        - 变异(Mutation): 位翻转、算术运算、有趣值替换、字典替换、拼接
        - 执行(Execute): fork子进程 → execv目标程序 → 写入测试用例到stdin
        - 测量(Measure): 读取共享内存位图 → 检查是否有新覆盖路径
        - 保留(Retain): 有新增代码覆盖 → 存入队列
      - 崩溃去重(Crash Dedup):
        - 所有崩溃保存在crashes/目录
        - AFL自动比较崩溃产生的覆盖位图，归类为unique crashes
    - AFL的关键文件结构：
      - 输入: corpus/ (种子队列目录)
      - 输出: out/ (Fuzzing结果目录)
        - out/queue/ — 产生新覆盖的所有测试用例
        - out/crashes/ — 触发崩溃的测试用例
        - out/hangs/ — 触发超时的测试用例
        - out/fuzzer_stats — 运行时统计信息
  参考文件：/home/a/成长为一个红队/archstrike-fuzz教学/01-模糊测试入门与AFL实战.txt

💻 第2小时 | 动手实践 (10:00 - 11:00)
  练习内容：AFL安装和第一个Fuzzing实验
  参考文件：/home/a/成长为一个红队/archstrike-fuzz教学/01-模糊测试入门与AFL实战.txt

  操作步骤：
    Step 1: 安装AFL和AFL++
      sudo apt install afl++
      或手动编译:
        git clone https://github.com/AFLplusplus/AFLplusplus.git /opt/AFLplusplus
        cd /opt/AFLplusplus && make
        sudo make install
      验证: afl-fuzz --help
    Step 2: 创建Fuzzing目标程序
      创建有漏洞的C测试程序:
        cat > /tmp/vulnerable.c << 'EOF'
        #include <stdio.h>
        #include <string.h>
        int main(int argc, char **argv) {
            char buf[32];
            FILE *f = fopen(argv[1], "r");
            if (!f) return 1;
            fread(buf, 1, 256, f);
            printf("Read: %s\n", buf);
            fclose(f);
            return 0;
        }
        EOF
      使用AFL编译:
        afl-gcc -o /tmp/vulnerable /tmp/vulnerable.c
      (注意: 缓冲区溢出漏洞 — buf[32] 但 fread 256)
    Step 3: 准备种子文件
      mkdir /tmp/afl_in /tmp/afl_out
      echo "AAAA" > /tmp/afl_in/seed.txt
      创建初始测试用例（种子）
    Step 4: 启动AFL Fuzzing
      sudo su -  (切换root，因为AFL需要设置核心转储配置)
      echo core > /proc/sys/kernel/core_pattern
      cd /tmp
      afl-fuzz -i afl_in/ -o afl_out/ -- ./vulnerable @@
      参数解释:
        -i: 输入目录(种子)
        -o: 输出目录(结果)
        @@: 用输入文件名替换
      AFL界面解读:
        process timing: 运行时间、最后路径发现时间、崩溃发现时间
        overall results: 总循环次数、总路径数、唯一崩溃数
        stage progress: 当前变异阶段和进度
    Step 5: 观察和分析AFL输出
      等待AFL发现崩溃(通常很快，因为程序有明显漏洞)
      查看崩溃目录:
        ls afl_out/crashes/
      cat afl_out/crashes/id:000000* → 查看触发崩溃的输入
      查看AFL统计:
        cat afl_out/fuzzer_stats
    Step 6: 分析崩溃原因
      使用GDB分析崩溃:
        gdb ./vulnerable
        (gdb) run < afl_out/crashes/id:000000*
        查看崩溃原因: SIGSEGV, 寄存器状态, 调用栈
      (gdb) bt → 获取backtrace
      (gdb) info registers → 查看是否EIP/RIP被覆盖
      记录崩溃分析结果

周二：Web模糊测试与AFL进阶
================================================================================

📚 第1小时 | 理论学习 (09:00 - 10:00)
  学习内容：Web模糊测试与AFL高级技巧
    - Web应用Fuzzing：
      - 与二进制Fuzzing的不同：
        - 输入是HTTP请求(参数、Header、Body)
        - 输出是HTTP响应(状态码、响应体、响应时间)
        - 需要代理将变异数据注入HTTP请求
      - Web Fuzzing工具：
        - fuff: 最快的Web Fuzzer，基于Go
        - wfuzz: Python Web Fuzzer
        - Burp Suite Intruder: 商用Web Fuzzer
        - gobuster: 目录Fuzzing
      - Web Fuzzing目标：
        - 目录/文件枚举
        - 参数名Fuzzing(发现隐藏参数)
        - 参数值Fuzzing(SQLi, XSS, Command Injection)
        - API端点枚举
    - AFL辅助工具：
      - afl-cov: 从AFL输出生成代码覆盖率报告
      - afl-plot: 生成覆盖率增长趋势图
      - afl-whatsup: 查看多个AFL实例的合并状态
      - afl-tmin: 最小化触发崩溃的测试用例
    - AFL字典(Dictionary)：
      - 创建关键词字典加速Fuzzing(针对特定输入格式)
      - 字典格式: 每行一个字符串
      - 如对XML输入: "<tag>", "</tag>", "attribute=", '"'
      - afl-clang-fast --dict=dict.txt编译时引用字典
    - 并行Fuzzing：
      - 多核心并行: afl-fuzz -M master & afl-fuzz -S slave1 & ...
      - 提高效率: 利用所有CPU核心
  参考文件：/home/a/成长为一个红队/archstrike-fuzz教学/02-协议与Web模糊测试.txt

💻 第2小时 | 动手实践 (10:00 - 11:00)
  练习内容：ffuf Web模糊测试与AFL进阶实验
  参考文件：/home/a/成长为一个红队/archstrike-fuzz教学/02-协议与Web模糊测试.txt

  操作步骤：
    Step 1: 安装ffuf Web Fuzzer
      sudo apt install ffuf
      或: go install github.com/ffuf/ffuf/v2@latest
      验证: ffuf -V
    Step 2: ffuf目录Fuzzing实战
      选一个测试目标(本地Web服务或授权测试目标):
        ffuf -u http://target/FUZZ -w /usr/share/seclists/Discovery/Web-Content/raft-medium-directories.txt -e .php,.html,.txt,.bak -o ffuf_dir_output.json
      参数:
        -u: URL, FUZZ是占位符
        -w: 字典文件
        -e: 扩展名列表
        -o: JSON格式输出
      分析结果: 关注状态码 200, 301, 403 的目录/文件
    Step 3: ffuf POST参数Fuzzing实战
      ffuf -u http://target/login.php -X POST -d "username=admin&password=FUZZ" -w /usr/share/seclists/Passwords/Common-Credentials/10k-most-common.txt -fc 401
      参数:
        -X POST: POST请求方法
        -d: POST数据
        -fc 401: 过滤掉401响应
      观察: 是否有返回非401的密码
    Step 4: AFL编译器对比（clang vs gcc）
      对比afl-gcc和afl-clang-fast:
        afl-clang-fast -o /tmp/vulnerable2 /tmp/vulnerable.c
      使用不同编译器各运行5分钟Fuzzing
      对比: 发现的路径数量、崩溃数量
      AFL++推荐使用afl-clang-fast: 更快的插桩，支持LLVM特性
    Step 5: AFL字典加速实验
      创建字典文件:
        cat > /tmp/dict.txt << 'EOF'
        "AAAA"
        "BBBB"
        "%s"
        "%n"
        "%x"
        "/bin/sh"
        "AAAAAAAABBBBBBBBCCCCCCCCDDDDDDDD"
        EOF
      使用字典: afl-fuzz -i afl_in/ -o afl_out_dict/ -x /tmp/dict.txt -- ./vulnerable @@
      对比有字典和无字典的崩溃发现速度差异
    Step 6: 使用afl-tmin最小化崩溃样本
      afl-tmin -i afl_out/crashes/id:000000* -o /tmp/minimized_crash -- ./vulnerable @@
      对比原始崩溃样本和最小化后的样本:
        xxd afl_out/crashes/id:000000*
        xxd /tmp/minimized_crash
      最小化帮助理解崩溃的根本原因

  练习内容：制定后续认证和进阶计划

  操作步骤：
    Step 1: 研究认证考试路径
      OSCP (Offensive Security Certified Professional):
        - 内容: 24小时实战考试 + 24小时报告
        - 前置: PEN-200课程, 60天实验室
        - 适合: 刚完成30周训练的你（基础已充分）
        - 准备时间: 3个月(PEN-200课程+实验室)
        - 费用: ~$1,599 (含课程和实验), 定期有折扣
      OSWE (Offensive Security Web Expert):
        - 内容: Web应用安全深入，白盒审计
        - 前置: 编程能力、Web安全经验
        - 适合: 对Web安全特别感兴趣的进阶路径
      OSEP (Offensive Security Experienced Penetration Tester):
        - 内容: 高级渗透和规避技术
        - 前置: OSCP推荐完成, 域和横向经验
        - 适合: 追求红队级能力的进阶路径
      CPTS (Hack The Box Certified Penetration Testing Specialist):
        - 性价比更高的替代选择
        - HTB Academy课程 + 考试
        - 偏向全流程渗透能力认证
    Step 2: 制定3个月短期计划（针对OSCP或其他认证）

      目标5: 参与Bug Bounty实战(补足Web安全深度的不足)
      目标6: 学习移动安全/云安全/工控安全等专项领域
