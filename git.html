安装 Git 下载与配置
Windows 安装（重点：添加到环境变量 PATH）
  1. 访问 git-scm.com/download/win 下载安装包
  2. 运行安装程序，关键步骤——在选择安装组件时，勾选 "Git from the command line and also from 3rd-party software"（将 Git 添加到 PATH）
  3. 在"Choosing the default editor"步骤选择你喜欢的编辑器（推荐 nano 或 VS Code）
  4. 在"Adjusting your PATH environment"步骤选择 "Git from the command line and also from 3rd-party software"（第二个选项）——这是最关键的一步，确保 Git 可被任意终端调用
  5. 其余步骤保持默认，完成安装
  6. 验证安装：打开 CMD 或 PowerShell，输入 git --version，应显示版本号
如果安装后终端提示"git 不是内部命令"，说明 PATH 未正确配置。手动添加：系统属性 → 高级 → 环境变量 → 系统变量 Path → 新建 → 添加 C:\Program Files\Git\bin 和 C:\Program Files\Git\cmd
Linux 安装： 
# Arch Linux
sudo pacman -S git

# Debian / Ubuntu
sudo apt install git

# Fedora
sudo dnf install git

# 验证
git --version
macOS 安装： 
# 通过 Homebrew（推荐）
brew install git

# 或安装 Xcode Command Line Tools（自带 git）
xcode-select --install

# 验证
git --version
全局初始配置（必须）： 
git config --global user.name "你的名字"
git config --global user.email "你的邮箱@example.com"
git config --global init.defaultBranch main
git config --global core.editor "nano"       # 或 "vim" / "code --wait"
git config --list                              # 查看所有配置
模块一 基础使用方法
Git 工作区域模型：工作区 → 暂存区(stage) → 本地仓库 → 远程仓库
git clone — 克隆远程仓库 
git clone https://github.com/user/repo.git          # 克隆到当前目录/repo
git clone https://github.com/user/repo.git my-folder  # 克隆到指定目录
git clone -b dev https://github.com/user/repo.git    # 克隆特定分支
git clone --depth 1 https://...                      # 浅克隆（只取最新版本，加速下载）
说明：clone 将远程仓库完整复制到本地，自动建立 origin 远程关联。--depth 1 仅下载最新提交历史，适合大型仓库快速获取代码。
git add — 暂存变更（stage 到暂存区） 
git add file.txt                # 暂存单个文件
git add file1.txt file2.txt    # 暂存多个文件
git add .                       # 暂存当前目录所有变更
git add -A                      # 暂存所有变更（包括删除的文件）
git add -p                      # 交互式逐块暂存（可选择性暂存每段修改）
说明：. 只暂存当前目录及其子目录。-A 暂存整个仓库的所有变更。-p 让你逐块审查代码，选择性地提交。
git commit — 提交到本地仓库 
git commit -m "提交信息"                                    # 带消息提交
git commit -am "提交信息"                                   # 暂存所有已跟踪文件并提交（跳过 git add）
git commit --amend -m "修正的提交信息"                     # 修改最后一次提交（不新增提交记录）
git commit --amend --no-edit                              # 追加暂存内容到上次提交，不改消息
说明：-a 自动暂存已跟踪的修改和删除（不包括新文件）。--amend 重写最后一次提交，适合修正 typo 或补充遗漏文件（注意：已推送的提交不要 amend）。
git push — 推送到远程仓库 
git push                                # 推送当前分支到 origin
git push origin main                    # 推送 main 分支到 origin
git push -u origin feature-branch       # 首次推送新分支并建立追踪关系
git push --force                        # 强制推送（覆盖远程历史，危险！）
git push --force-with-lease            # 安全强制推送（远程有新提交时拒绝，推荐）
说明：-u(set-upstream) 将本地分支与远程分支关联，之后只需 git push。--force-with-lease 比 --force 安全，它检查远程是否有你不知道的新提交。
git pull — 拉取并合并远程更新 
git pull                          # 拉取并合并当前分支
git pull origin main              # 拉取指定远程分支
git pull --rebase                 # 拉取后变基（避免多余合并提交，推荐）
git fetch                         # 仅拉取远程信息不合并
git fetch origin                  # 拉取所有远程分支的最新信息
说明：pull = fetch + merge。--rebase 用变基代替合并，保持提交历史线性整洁。fetch 只下载远程更新不合并，适合先查看再决定。
git status — 查看当前状态 
git status                        # 查看工作区和暂存区状态
git status -s                     # 简略输出（M=已修改, ??=新文件, A=已暂存）
git status -sb                    # 简略输出 + 分支信息
git config — 配置管理 
git config --global user.name "Name"     # 全局级别（~/.gitconfig）
git config --local user.email "a@b.com"  # 仓库级别（.git/config）
git config --system core.editor vim      # 系统级别（/etc/gitconfig）需管理员
git config --list                         # 列出所有配置
git config --global --unset user.name     # 删除配置项
优先级：local > global > system。仓库级覆盖全局，全局覆盖系统。
git log — 查看提交历史 
git log                                  # 完整提交历史
git log --oneline                        # 每提交一行简略显示
git log --oneline --graph --all          # 彩色分支图（最常用）
git log -p                                # 显示每次提交的具体改动(diff)
git log --author="名字"                   # 按作者筛选
git log --since="2026-01-01"             # 按日期筛选
git diff — 查看差异 
git diff                        # 工作区 vs 暂存区（未暂存的修改）
git diff --staged               # 暂存区 vs 最新提交
git diff HEAD                   # 工作区 vs 最新提交（所有未提交的修改）
git diff main..feature          # 比较两个分支的差异
git branch / checkout / switch — 分支管理 
git branch                            # 列出本地分支（* 表示当前）
git branch -a                         # 列出所有分支（含远程）
git branch feature-login              # 创建新分支
git branch -d feature-login           # 删除已合并的分支
git branch -D feature-login           # 强制删除分支（不管是否合并）
git switch feature-login              # 切换到分支（Git 2.23+推荐）
git switch -c feature-login          # 创建并切换到新分支
git checkout feature-login            # 切换分支（传统方式）
git checkout -b feature-login         # 创建并切换（传统方式）
说明：switch 是 Git 2.23 引入的更安全的分支切换命令，职责单一。checkout 功能更杂（也可恢复文件），建议用 switch。
git merge / rebase — 合并分支 
git merge feature-login              # 将 feature-login 合并到当前分支
git merge --no-ff feature-login      # 禁用快进，保留分支历史
git rebase main                       # 将当前分支变基到 main 之上
git rebase -i HEAD~3                 # 交互式变基最近 3 个提交（合并/编辑/重排/删除）
merge vs rebase：merge 保留完整分支历史，适合公共分支。rebase 将提交"搬家"到目标分支顶部，生成线性历史，适合个人分支整理。
🔑 GitHub SSH 密钥配置（git push 免密推送）
配置 SSH 密钥后，每次 git push 无需输入密码，是 GitHub 推荐的认证方式。
步骤 1：在本地生成 SSH 密钥对 
# 生成 ED25519 密钥（推荐，更安全更快）
ssh-keygen -t ed25519 -C "你的邮箱@example.com"
# 一路回车：
# Enter file: 直接回车（使用默认路径 ~/.ssh/id_ed25519）
# Enter passphrase: 直接回车（不设密码）或输入密码

# 如果系统不支持 ED25519，用 RSA：
ssh-keygen -t rsa -b 4096 -C "你的邮箱@example.com"

# 查看公钥内容（需要复制它）
cat ~/.ssh/id_ed25519.pub
# 输出类似：ssh-ed25519 AAAAC3NzaC1l... 你的邮箱@example.com
步骤 2：将公钥添加到 GitHub 账号
  1. 复制 ~/.ssh/id_ed25519.pub 的全部内容
  2. 打开 GitHub → 右上角头像 → Settings
  3. 左侧菜单 → SSH and GPG keys
  4. 点击 New SSH key
  5. Title 填一个辨识名称（如 "我的笔记本"），Key 粘贴公钥内容
  6. 点击 Add SSH key
步骤 3：验证连接 
ssh -T git@github.com
# 成功输出：Hi 你的用户名! You've successfully authenticated...
之后使用 SSH 地址克隆和推送（git@github.com:用户名/仓库.git）即无需密码。不要在 HTTPS 地址中硬编码密码或 token。
📦 传输大型文件 — Git LFS（Large File Storage）
普通 Git 不适合管理大于 100MB 的文件（每次克隆都下载完整历史，仓库迅速膨胀）。Git LFS 将大文件替换为指针，实际文件存储在独立服务器上。
场景：你的项目里有一个 1.5GB 的数据库备份文件 data/dump.sql 需要版本管理。
安装 Git LFS： 
# Linux（Arch）
sudo pacman -S git-lfs

# Linux（Debian/Ubuntu）
sudo apt install git-lfs

# macOS
brew install git-lfs

# 初始化（每个用户只需执行一次）
git lfs install
跟踪大文件并上传： 
# 进入你的项目目录
cd my-project

# 告诉 Git LFS 哪些文件类型用 LFS 管理
git lfs track "*.sql"           # 所有 SQL 文件
git lfs track "*.zip"           # 所有 ZIP 文件
git lfs track "*.psd"           # 所有 Photoshop 文件
git lfs track "data/*.bin"      # data 目录下的二进制文件

# 这会生成/更新 .gitattributes 文件，需要提交它
git add .gitattributes
git commit -m "配置 Git LFS 跟踪规则"

# 正常添加大文件（Git LFS 自动接管）
git add data/dump.sql
git commit -m "添加数据库备份（1.5GB，使用 LFS）"
git push origin main
# 推送时大文件上传到 LFS 服务器，仓库本体只存指针
其他人克隆时： 
# 普通 clone 即可，LFS 文件自动下载
git clone https://github.com/user/repo.git

# 如果 LFS 文件没下载（离线或跳过），手动拉取：
git lfs pull

# 查看 LFS 管理的文件列表
git lfs ls-files
Git LFS vs 替代方案对比： 
┌────────────┬──────────────────┬─────────────────────────────┐
│   方案     │     适用场景     │          优缺点             │
├────────────┼──────────────────┼─────────────────────────────┤
│ Git LFS    │二进制文件<2GB │ GitHub 免费 1GB 存储/月     │
│ git-annex  │ 海量大文件管理   │ 自托管、更灵活、学习曲线陡  │
│ .gitignore │ 不需要版本管理   │ 最简单，但文件不同步到其他人│
│ 外部存储   │ 巨型文件 >5GB │ 用网盘/S3，README 里放链接  │
└────────────┴──────────────────┴─────────────────────────────┘
模块二 个人使用详解（完整实战）
本模块以完整示例演示：终端登录 GitHub → 创建仓库 → 上传项目 → 修改更新 → 提交推送的全流程。
步骤 1：在终端通过 Git 登录 GitHub
可通过 SSH 密钥（推荐）或 Personal Access Token（PAT）两种方式。
方法① — SSH 密钥登录（推荐，一次配置永久使用）： 
# 生成 SSH 密钥（一路回车即可）
ssh-keygen -t ed25519 -C "你的邮箱@example.com"

# 查看公钥并复制
cat ~/.ssh/id_ed25519.pub
将公钥内容粘贴到 GitHub → Settings → SSH and GPG keys → New SSH key。
方法② — Personal Access Token 登录（适合临时或不支持 SSH 的环境）：
GitHub → Settings → Developer settings → Personal access tokens → Tokens(classic) → Generate new token。
勾选 repo 权限，生成 token。
之后在终端操作时，用户名填你的 GitHub 用户名，密码填生成的 token 字符串。
步骤 2：在 GitHub 创建远程仓库 + 本地关联
假设你要创建一个名为 "my-website" 的项目。 
# 在 GitHub 网页上点击 + → New repository → 仓库名: my-website → 不勾选 README

# 本地创建项目
mkdir my-website
cd my-website
echo "# My Website" > README.md
echo "<h1>Hello World</h1>" > index.html

# 初始化 Git
git init
git add .
git commit -m "首次提交：创建网站骨架"

# 关联远程仓库并推送
git remote add origin git@github.com:你的用户名/my-website.git
git branch -M main
git push -u origin main
说明：git remote add origin 将远程仓库命名为 origin（惯例名称）。git branch -M main 将默认分支重命名为 main。-u 建立追踪关系，后续只需 git push。
🖥 纯终端创建仓库（无需打开网页）— 适用于无桌面 Linux 环境
如果你正在一台没有图形界面的服务器上工作，可以通过以下两种方法完全在终端中创建 GitHub 远程仓库。
方法① — 使用 GitHub CLI（推荐，最简洁）： 
# 安装 gh（GitHub 官方命令行工具）
sudo pacman -S github-cli          # Arch Linux
sudo apt install gh                # Debian/Ubuntu

# 首次使用需登录（按提示选择 SSH 认证）
gh auth login

# 在已有项目的文件夹中直接创建 GitHub 仓库并推送
cd ~/my-existing-project           # 进入已写好代码的文件夹
git init
git add .
git commit -m "首次提交"

# 一行命令：创建 GitHub 仓库 + 关联 + 推送
gh repo create my-project --private --source=. --remote=origin --push
# --private  创建私有仓库（改 --public 为公开）
# --source=. 以当前目录为源
# --push    直接推送代码
方法② — 使用 GitHub API + curl（无需安装额外工具）： 
# 创建 Personal Access Token（如果还没有）
# GitHub → Settings → Developer settings → Tokens → Generate
# 勾选 repo 权限 → 生成后复制 token 字符串

# 导出 token 为环境变量
export GITHUB_TOKEN="ghp_xxxxxxxxxxxxxxxxxxxx"

# 通过 API 创建仓库
curl -H "Authorization: token $GITHUB_TOKEN" \
     -H "Accept: application/vnd.github+json" \
     https://api.github.com/user/repos \
     -d '{"name":"my-project","private":true}'

# 关联远程仓库并推送
git init && git add . && git commit -m "首次提交"
git remote add origin git@github.com:你的用户名/my-project.git
git branch -M main
git push -u origin main
推荐方法① gh，一条命令完成创建+关联+推送。方法②适合无法安装额外软件的极简环境。无论哪种方法，都无需打开浏览器。
步骤 3：修改项目并更新
假设你要修改网站内容并添加新页面。 
# 编辑文件
echo "<h2>Welcome to my site</h2>" >> index.html

# 添加新文件
echo "body { background: #f0f0f0; }" > style.css

# 查看变更
git status
# modified:   index.html
# Untracked:  style.css

# 暂存并提交
git add index.html style.css
git commit -m "更新首页内容并添加样式文件"

# 推送到 GitHub
git push
步骤 4：版本回退和恢复 
# 查看提交历史
git log --oneline
a1b2c3d 更新首页内容并添加样式文件
e4f5g6h 首次提交：创建网站骨架

# 回退到上一个版本（保留工作区修改）
git reset --soft HEAD~1                # 撤销提交，但保留暂存区和工作区

# 回退并清空暂存区（修改留在工作区）
git reset HEAD~1                        # 撤销提交和暂存，修改留在工作区

# 完全回退（丢弃所有修改，危险！）
git reset --hard HEAD~1

# 撤销未暂存的修改
git checkout -- index.html              # 丢弃 index.html 的未暂存修改
git restore index.html                  # 新版推荐写法（Git 2.23+）
git reflog 记录了你所有的 HEAD 移动历史。即使执行了 reset --hard，也能通过 git reflog 找回"丢失"的提交。
模块三 多人协作详解
Git 的核心价值在于多人协作。本模块分两个层次：参与他人项目（贡献者视角）和组织团队项目（管理者视角）。
一、参与别人的项目（Fork + Pull Request 流程）
  1. 在 GitHub 上 Fork 目标仓库（点击仓库右上角 Fork 按钮）→ 你的账户下出现同名仓库
  2. git clone git@github.com:你的用户名/项目名.git — 克隆你 Fork 的仓库到本地
  3. git switch -c fix-typo — 创建新分支描述你的修改意图
  4. 修改代码 → git add . && git commit -m "修正 README 中的拼写错误"
  5. git push -u origin fix-typo — 推送到你的 Fork 仓库
  6. 回到 GitHub → 你的 Fork 仓库页面 → 点击 "Contribute" → "Open Pull Request"
  7. 撰写 PR 描述 → 提交 → 等待原仓库维护者审核合并
PR 被合并后，你的 GitHub 用户名会出现在原仓库的 Contributors 列表中。
查看别人的提交：git log --author="对方名字" 或 git show 提交SHA 查看具体改动。
二、团队协作——分支策略与工作流程
主流分支模型（GitHub Flow）：
  • main 分支 — 始终可部署的稳定代码，禁止直接推送
  • feature/* 分支 — 每个新功能/修复从 main 分出独立开发
  • 完成后发起 Pull Request → 团队 Code Review → 合并回 main
多人协作常用命令： 
# 查看所有远程分支
git branch -a

# 拉取别人的分支到本地
git fetch origin
git switch -c review-branch origin/同事的分支名

# 查看别人提交了什么
git log origin/同事的分支名 --oneline --graph
git diff main..origin/同事的分支名        # 查看与 main 的差异

# 解决冲突
git merge main                           # 将 main 合并到当前分支触发冲突
# 手动编辑冲突文件 → 保留正确代码 → 删除冲突标记
git add .
git commit -m "解决合并冲突"

# 撤销错误合并
git merge --abort                        # 放弃当前合并
git reset --hard HEAD~1                  # 回退到合并前（未推送时可用）
三、多人合作实战模拟（五人团队开发"喵记笔记"应用）
以下模拟一个 5 人团队从零开始协作开发的全过程。团队角色：
小明（项目 Leader，稳健派）、小红（前端，急性子）、小刚（后端，沉默实干）、小美（全栈，细节控）、小强（新人，爱提问）
阶段一：项目初始化（小明创建仓库并邀请队友） 
小明在 GitHub 创建仓库 "meow-notes"，添加 README 和 .gitignore
git clone 到本地，创建项目骨架（目录结构、package.json 等）
git add . && git commit -m "初始化项目骨架"
git push

小明在 GitHub → Settings → Collaborators → 添加小红、小刚、小美、小强为协作者
（或者：团队成员各自 Fork 仓库，通过 PR 协作）
阶段二：并行开发（每人独立分支工作） 
小红（前端）→ git clone → git switch -c ui-login-page
  → 开发登录页面 HTML/CSS → commit → push → 发起 PR

小刚（后端）→ git switch -c api-auth
  → 编写用户认证 API → commit → push → 发起 PR

小美（全栈）→ git switch -c database-schema
  → 设计数据库表结构 → commit → push → 发起 PR
阶段三：Code Review 与合并（小明统筹审查）
小明：小美的数据库 schema 写得很规范，我先合了。小红，你的 CSS 里用了绝对定位，在移动端会错位，改成 flexbox 吧。
小红：好，我改成 flex 布局，再推一版。
小刚：我的 API 分支和小美的 schema 有依赖，需要等她合并后我 rebase 一下。
小美：我的已合并，小刚你可以 rebase 了。
小刚：git switch api-auth && git rebase main → 解决一处冲突 → push。
阶段四：新人加入（小强的学习路径） 
小强 git clone 项目 → 阅读 README 和已有代码
小强 git switch -c fix-readme-typo → 发现 README 有错别字
  → 修改 → git add && git commit -m "修正 README 拼写"

阶段五：冲突解决（团队协作的日常）
小红：我改了 App.css 第 42 行附近，推上去了。
小美：等等我也改了同一块区域！我刚 push 被拒绝了。
小明：@小美 先 git pull --rebase，然后解决冲突，再 git push。
小美：git pull --rebase → CONFLICT in App.css → 打开文件 → 看到 <<< === >>> 标记 → 保留我和小红各自正确的部分 → git add → git rebase --continue → git push。搞定了！
团队协作最佳实践总结： 
# 1. 每天开始工作前：同步最新代码
git switch main && git pull
git switch 你的分支 && git rebase main

# 2. 提交信息格式规范（Conventional Commits）
feat: 添加用户登录功能
fix: 修复移动端布局错位
docs: 更新 API 文档
refactor: 重构数据库查询逻辑

# 3. 一个分支只做一件事
#  错误：ui-login 分支同时改了登录页面 + 数据库结构 + 文档
#  正确：ui-login 只改登录页面，database 只改数据库

# 4. 推送前先拉取（避免冲突累积）
git pull --rebase origin main && git push

# 5. Code Review 礼仪
#  - 审查别人代码时关注逻辑而非风格（风格交给 linter）
#  - 提出问题而非命令（"这里是否考虑边界情况？"而非"这里写错了"）
#  - 及时 review（不超过 24 小时）
管理者视角——如何统筹多人项目：
• 在 GitHub 仓库 Settings → Branches → 添加分支保护规则：要求 PR 审查、要求状态检查通过后才能合并到 main
• 使用 GitHub Issues 分配任务，关联 commit（commit message 写 fixes #12 自动关闭 Issue）
• 使用 GitHub Projects 看板管理任务状态（Todo → In Progress → Review → Done）
• 定期 git log --all --oneline --graph 查看全貌，确保无人偏离方向
• Squash Merge：将 PR 的多个小提交合并为一个整洁的提交再合入 main
掌握了这些 Git 技能，你就可以自信地参与任何开源项目了！
